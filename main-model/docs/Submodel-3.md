# Submodel-3 模块（NFC 读卡）

## 概述

本模块为近场通信扩展方案，用于读取 NFC 标签、门禁卡、公交卡等，实现身份识别与数据交换。

- **主控芯片**：CH32V103C8T6
- **与核心模块接口**：模块侧 UART1，**115200/8-N-1**
- **模块 ID**：`0x40` ~ `0x42`
- **模块类型编号**：`0x05`
- **子类型编号**：`0x03`（NFC）

## 硬件组成

- **CH32V103C8T6**：NFC 芯片控制、协议帧转发、卡数据管理
- **NFC 读写芯片**：如 PN532（I2C/SPI/UART）、RC522（SPI）或 ST25R 系列
- **NFC 天线**：PCB 环形天线或外接铁氧体天线，13.56 MHz
- **LED 指示（可选）**：读卡成功 / 失败状态灯

## 软件职责

### 初始化阶段

- 初始化 CH32V103 时钟、GPIO、I2C/SPI/UART（根据 NFC 芯片选型）、UART1（115200）
- 初始化 NFC 芯片：配置工作模式（读卡器模式）、射频功率、检测阈值
- 初始化本地卡片白名单 / 记录缓存（可选，存于 Flash）
- 向 Core 回复 `CMD_GET_TYPE`，上报子类型 `0x03`

### 运行阶段

- **卡片轮询**：定时发送寻卡指令（如 ISO14443A 的 REQA / WUPA），检测天线场内是否有卡片进入
- **数据读取**：
  - 卡片靠近后，执行防冲突、选卡、认证（若 MIFARE Classic 需密钥认证）
  - 读取 UID、扇区数据或 NDEF 消息
- **事件上报**：主动通过 UART1 向 Core 上报：
  - 识别成功：携带 UID、卡类型、读取到的数据长度及内容
  - 识别失败：如认证失败、不支持的卡类型、射频通信错误
- **指令响应**：
  - 响应 Core 的写卡指令：向可写标签写入 NDEF 记录或自定义数据
  - 响应配置指令：修改射频功率、切换卡类型支持（Type A / Type B / FeliCa）
- **记录管理（可选）**：缓存最近 N 次读卡记录，供 Core 查询历史

### 中断与事件

- NFC 芯片中断（如有）：卡片进入 / 离开天线场检测
- 定时器中断：周期性寻卡轮询（建议 100~300 ms 间隔，平衡响应速度与功耗）
- UART1 中断：接收 Core 的读写 / 配置指令

## 命名规范

- **文件**：统一采用下划线加小写字母，例如 `hardware.c`、`nfc.h`
- **函数**：每个单词首字母大写加下划线，例如 `Hardware_Init()`、`Nfc_Poll()`
- **变量**：统一小写加下划线，例如 `card_uid`、`nfc_state`
- **文件夹**：下划线写法，仅首字母大写，例如 `Nfc`、`Flash`
- **系统目录**（`Core`、`Debug`、`Peripheral`、`Startup`）保持不动，不参与上述命名规范

## 开发要点

1. **天线设计**：13.56 MHz NFC 天线对谐振频率敏感，PCB 绘制时需按芯片手册计算电感线圈的圈数、线宽、间距，必要时用网络分析仪调谐；天线面积越大读卡距离越远，但受模块体积限制。
2. **卡片兼容性**：PN532 支持 Type A / Type B / FeliCa，兼容性较好；RC522 仅支持 Type A（MIFARE），成本低但兼容性弱。建议根据目标场景（门禁卡多为 MIFARE，手机 NFC 多为 Type A/B）选型。
3. **密钥安全**：若读写 MIFARE Classic 加密扇区，密钥不可硬编码在明文固件中；至少做简单混淆或存储在 Flash 中允许用户通过 Core 配置。
4. **防抖动**：卡片在天线边缘可能反复进入 / 离开检测范围，软件需做去抖（如连续 2~3 次检测到同一 UID 才认定为有效刷卡），避免频繁误上报。
5. **与 Core 的协同**：NFC 标签数据（如 URL、vCard）的解析与 UI 展示由 Core / Display 负责；Submodel-3 只负责底层射频通信与 UID / 原始数据读取。

# Submodel-1 模块（指纹识别）

## 概述

本模块为生物识别扩展方案，用于用户身份验证、解锁终端及支付授权等场景。

- **主控芯片**：CH32V103C8T6
- **与核心模块接口**：模块侧 UART1，**115200/8-N-1**
- **模块 ID**：`0x40` ~ `0x42`（根据插入的物理槽位分配）
- **模块类型编号**：`0x05`
- **子类型编号**：`0x01`（Fingerprint）

## 硬件组成

- **CH32V103C8T6**：指纹模块控制、协议帧转发、模板管理
- **指纹传感器**：光学或电容式指纹采集模组（如 AS608、R307 等主流型号），通常通过 UART 或 SPI 与主控通信
- **Flash 存储（可选）**：若传感器无内部大容量存储，可外接 SPI Flash 存放指纹模板数据

## 软件职责

### 初始化阶段

- 初始化 CH32V103 时钟、GPIO、UART1（115200）、与指纹传感器的通信接口
- 初始化指纹传感器：软复位、验证通信、读取已注册指纹数量
- 初始化本地指纹模板存储表（若在本地 Flash 中管理）
- 向 Core 回复 `CMD_GET_TYPE`，上报子类型 `0x01`

### 运行阶段

- **指纹采集与比对**：
  - 传感器检测到手指按压后，采集指纹图像并提取特征点
  - 与本地模板库比对，输出匹配结果（ID + 匹配度）或比对失败
- **事件上报**：主动通过 UART1 向 Core 发送识别结果：
  - 识别成功：`CMD_SUB_EVT_DATA` / `CMD_DISP_EXT_FP_STATUS` 格式，携带指纹 ID 与匹配度
  - 识别失败：上报失败事件，可附带失败原因（如未注册、图像质量差）
- **指令响应**：
  - 响应 Core 的注册指纹指令：采集图像 → 生成模板 → 分配 ID → 存入 Flash
  - 响应删除指纹指令：按 ID 或全部清空
  - 响应查询指令：返回已注册数量、模板 ID 列表
- **安全策略**：连续识别失败多次（如 5 次）后，暂时锁定并上报 Core，由 Core 决定是否延时解锁或触发告警

### 中断与事件

- 指纹传感器中断（如有）：手指按压检测（通常传感器自带触摸输出引脚）
- UART1 中断：接收 Core 的注册 / 删除 / 查询指令
- 定时器中断：周期性保活心跳或传感器状态巡检

## 命名规范

- **文件**：统一采用下划线加小写字母，例如 `hardware.c`、`fingerprint.h`
- **函数**：每个单词首字母大写加下划线，例如 `Hardware_Init()`、`Fp_Enroll()`
- **变量**：统一小写加下划线，例如 `fp_id`、`match_score`
- **文件夹**：下划线写法，仅首字母大写，例如 `Fingerprint`、`Flash`
- **系统目录**（`Core`、`Debug`、`Peripheral`、`Startup`）保持不动，不参与上述命名规范

## 开发要点

1. **指纹传感器选型**：优先选择带内部 DSP 和 Flash 的模组（如 AS608），主控只需发送指令和读取结果，无需在 CH32V103 上运行指纹识别算法，大幅降低开发难度。
2. **模板管理**：若传感器自带存储（通常支持 100~300 枚指纹），可直接使用；若需扩展，可通过 SPI Flash 存放模板，但需自行维护索引表和磨损均衡。
3. **识别速度**：光学传感器通常需要 0.5~1.5 秒完成采集+比对，建议在识别期间点亮状态指示灯（如模块上的 LED），给用户明确反馈。
4. **隐私安全**：指纹模板数据不应以明文形式通过 UART 传输；若 Core 请求读取原始模板（如备份），建议做简单异或或加密处理。
5. **与 Core 的协同**：指纹注册流程通常需要 UI 引导（提示用户按多次手指），因此注册指令一般由 Core 发起，模块执行采集与存储，过程中分步上报进度（如“请按第一次”、“请按第二次”、“注册成功”）。
